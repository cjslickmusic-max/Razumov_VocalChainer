# Критический обзор кода (аудит)

Документ фиксирует **замечания по качеству, рискам и архитектуре** без претензии на полноту. Приоритет: realtime, граф, состояние.

**Дата:** 2026-03-28

---

## 1. Аудиопоток и realtime

### 1.1 Аллокации и блокировки

- **`GraphEngine::swapAndPreparePendingPlan`** берёт `std::mutex` на **каждом** `process` (аудиопоток). Swap указателя на план короткий, но при конкуренции с UI возможны задержки и **priority inversion** в тяжёлых хостах. Типичный паттерн — lock-free указатель / double buffer без mutex в audio, либо минимизация работы под замком.
- **`GraphEngine::processSplit`** (исправлено): ранее **`std::vector<int>` на каждый вызов** давал аллокацию кучи в горячем пути при каждом split и каждом аудиоблоке. Сейчас используется предвыделенный `latPerBranchScratch_`, размер задаётся в `ensureBranchPool` (при `prepare`).
- **`SpectralCompressorNode::process`**: `dryScratch_.setSize(ch, n, ...)` при смене `n` между вызовами без нового `prepare` теоретически может трогать кучу. Обычно хост держит стабильный block size после `prepareToPlay`; при смене буфера JUCE вызывает `prepare` снова — всё же стоит явно задокументировать контракт или держать scratch по `maxBlockSize_` без изменения размера в `process`.

### 1.2 Обход графа каждый блок

- **`walkApplyPhase3Segment` / `walkApplyPhase3Slot`**: полный DFS по дереву слотов на **каждый** блок для применения параметров. Сложность **O(узлы)** на блок; при очень больших графах — заметный overhead. Возможные направления: кэш «грязных» слотов, плоский список активных модулей, снимок параметров раз в блок без обхода дерева (отдельная задача).

### 1.3 Поиск параметров по слоту

- **`ModuleParamsRuntime::findSlotIndex`**: линейный поиск по `slotIds_` при каждом `fillSlot`. При десятках слотов ещё терпимо; при сотнях — рассмотреть `unordered_map<uint32_t, size_t>` или сортированный массив + бинпоиск (после согласования с message thread).

## 2. DSP и узлы

### 2.1 Spectral compressor

- Большие буферы **`ChannelData::olaSize = 65536`**, векторы на канал — память и кэш; для продукта иметь в виду **измерения CPU** (`docs/PERF_CHECKPOINTS.md`).
- Логика **wet/dry** и **PDC** внутри узла согласована с продуктовой моделью; при добавлении **phase align** между ветками графа не дублировать конфликтующую политику с внутренним dry/wet spectral.

### 2.2 Mic correction

- Узел по смыслу **pass-through**; в коде зафиксирован **TODO** на EQ/IR/профиль — ожидаемо по roadmap (микрофон в конце приоритетов).

### 2.3 Компрессоры

- Отдельные узлы по архетипам — хорошо для продукта. Качество моделей — **отдельная долгая** линия работ (субъективно + измерения), не блокер для инфраструктуры графа.

## 3. Состояние, UI, сериализация

- **FlexGraph + ModuleParams** в APVTS: модель согласована с handoff; при добавлении полей — миграции и тесты загрузки старых проектов.
- **Редактор**: инспектор показывает **полный** набор ручек фазы 3 даже если в **Compact** цепочке часть узлов отсутствует — UX «мертвые» контролы до смены профиля задокументирован в ROADMAP; при полировке можно прятать неиспользуемое.
- **Макросы**: пересечения параметров (например LP у нескольких макросов) — осознанный компромисс; при тюнинге проверять лимиты и поведение.

## 4. Тесты и инфраструктура

- **`RazumovVocalChainTests`**: хорошее покрытие merge/PDC/вложенных split; **нет** автоматического запуска в CI в репозитории (по состоянию на дату аудита) — см. фаза 5 roadmap.
