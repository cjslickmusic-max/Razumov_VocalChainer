# FFT and spectrum engineering (VocalChainer)

**When to read this:** любая задача с **спектром, FFT, анализатором полосы**, визуализацией после узла графа, или оптимизацией **SIMD/векторизации** для анализа.

**Связанные правила:** `.cursor/rules/dsp-audio-safety.mdc`, `vocal-chain-graph-dsp.mdc`, `node-module-conventions.mdc`.

**Человеческая выжимка:** `docs/FFT_SPECTRUM_ARCHITECTURE.md`.

---

## 1. Цель

- Везде, где показываем **FFT/спектр**, используем **один согласованный контракт**: откуда берётся сигнал, как часто считаем, как передаём в UI.
- **После каждого прибора** спектр «следующего экрана» = **сигнал после этого узла** (или явно выбранная точка тапа). Нет «магического одного FFT на всё» без привязки к графу.

---

## 2. Два потока (обязательное разделение)

| Место | Роль |
|-------|------|
| **Audio thread** | Только: копия/кольцевой буфер фиксированного размера, **один** FFT размера N на блок (или реже — см. ниже), запись **готовых** бинов/энергий в lock-free или atomic-снимок. **Никаких** аллокаций, файлов, тяжёлых mutex в `process`. |
| **Message / UI thread** | Чтение снимка, логарифмическая ось, сглаживание, перерисовка, выбор «показать после слота X». |

Перенос «тяжёлого» FFT в фоновый поток допустим только при **жёсткой** модели: очередь готовых кадров, без блокировки audio thread.

---

## 3. Общая функция анализа (контракт вызова)

**Идея:** одна реализация «проанализировать кадр» (окно, FFT, magnitude/power), вызываемая из **одного места** в движке или из узла-тапа, а не копипаста в каждом модуле.

**Алгоритм вызова (политика):**

1. **Точка тапа:** для каждого места, где нужен спектр (после EQ, после фильтра, «глобальный» инспектор), задаётся **идентификатор тапа** = узел/слот + опционально «pre/post».
2. **Когда считать:** в конце `process` этого узла (или сразу после `process` в движке для вставленного тапа), если накоплено >= hop сэмплов (или раз в N блоков — см. нагрузку).
3. **Вход:** моно или L+R по политике продукта (документировать: «метр L», «mid», и т.д.).
4. **Выход:** нормализованные бины или dB в **заранее выделенный** буфер, затем копия в атомики / lock-free слот для UI.

**Пересчёт «после предыдущего узла»:** не один глобальный FFT на весь буфер хоста, а **цепочка**: тап на выходе узла A даёт спектр после A; тап после B — после B. Параллельные ветки: отдельный тап/снимок на ветку или явное «merge point only».

---

## 4. Частотная дисциплина

- `binCenterHz(k) = k * sampleRate / fftSize` (или эквивалент для half-complex); **Nyquist** не превышать.
- Логарифмическая ось и «умные» полосы — **только в UI** (или в отдельной таблице пересчёта), не смешивать с сырыми бинами без документации.
- `prepare` фиксирует `sampleRate` и размер FFT; смена SR — сброс планов и таблиц.

---

## 5. SIMD и векторизация (задел вперёд)

Писать анализ так, чтобы потом **не переписывать граф** с нуля:

- **Буферы:** выравнивание под SIMD (например 16/32 байт) для массивов, где будет `float`/`double` векторные циклы; один непрерывный блок на кадр.
- **Магнитуда/постобработка:** циклы `for (k)` по бинам — кандидат на **ручную векторизацию** или компилятор с `-O3`; избегать непредсказуемых веток внутри горячего цикла.
- **FFT:** использовать **одну** библиотечную реализацию (например JUCE `dsp::FFT` / платформенные vDSP/FFTW на усмотрение лицензии). Не плодить несколько FFT разных размеров без нужды.
- **План:** горячий путь = окно + FFT + `|X|^2` или magnitude; **не** смешивать с логикой UI.

Платформенные intrinsic (AVX/SSE/NEON) — только в **узких** файлах-обёртках или за `#if` с **одинаковой** семантикой для macOS/Windows (см. `cross-platform-only.mdc`).

## 6. Чеклист для нового фича со спектром

- [ ] Где точка тапа в графе (слот/узел)?
- [ ] Моно/стерео/какой канал?
- [ ] Размер FFT и hop; допустимая частота обновления UI (30 Hz и т.д.)?
- [ ] Нет аллокаций в audio thread при стабильной работе?
- [ ] UI читает только снимок; сброс при смене SR/графа?
- [ ] Нужно ли добавить строку в `docs/PERF_CHECKPOINTS.md` после замеров?

---

## 7. Антипаттерны

- FFT в `paint()` или `timerCallback` с чтением **прямого** буфера процессора без синхронизации.
- Разный код FFT в каждом модуле «чуть-чуть по-другому».
- Подмена «спектр после цепочки» одним FFT без явной привязки к узлу.

## 8. Связь с кодом репозитория

При появлении общего хелпера — держать его в одном месте (например `dsp/analysis/` или существующий `SpectrumTap`), подключать узлы через `ISpectrumSource` / фабрику графа. Обновлять `docs/ARCHITECTURE.md` и этот skill при смене контракта.
