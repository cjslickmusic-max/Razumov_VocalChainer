# FFT / spectrum: архитектура и политика

Краткий документ для людей; **полный чеклист и правила для агентов** — в Cursor skill:

**`.cursor/skills/fft-spectrum-engineering/SKILL.md`**

## Зачем

Во многих местах UI (EQ, фильтр, анализатор, спектральные модули) нужен **один способ** думать о FFT: откуда сигнал, как часто считаем, как передаём в UI, как не ломать real-time поток.

## Принципы

1. **Разделение потоков:** тяжёлый расчёт и запись снимка — audio thread по правилам `dsp-audio-safety.mdc`; отрисовка и лог-ось — UI.
2. **Где считаем:** спектр «после узла N» = тап **на выходе** этого узла; не один неоговорённый глобальный FFT.
3. **Общая функция:** одна реализация «окно + FFT + magnitude/energy» (или семейство с фиксированными N), вызываемая из движка/тапа по согласованному контракту.
4. **SIMD:** буферы и горячие циклы по бинам готовить к векторизации; FFT — одна библиотечная реализация; платформенные оптимизации — в изолированных обёртках.

## Связанные файлы

- `SpectrumTap`: FFT `kFftSize` (обычно 4096); **256 display-бинов по логарифмической оси 20 Hz…min(20 kHz, Nyquist)** (не равные срезы по индексу FFT — иначе низ теряет гармоники); амплитуда **нормируется на `fftSize`** перед `20*log10`, иначе уровень «в небо» и всё упирается в верх графика; UI EQ — два слоя (`spectrumDisplay_` + peak-decay `spectrumTrail_`).
- Узлы/тапы: `Source/dsp/graph/` (например `SpectrumTap`, `SpectrumAnalyzerNode`, EQ).
- Правила: `.cursor/rules/dsp-audio-safety.mdc`, `cross-platform-only.mdc`.
